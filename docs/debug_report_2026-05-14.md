## 1. ChromaDB Query Syntax Error

**Severity:** High — breaks RAG experience retrieval  
**Location:** `skyn3t/rag/vector_store.py` (line 128) and `skyn3t/agents/code_agent.py` (line 961)

### Symptom
Log entries show:
```
chroma query failed: Expected where to have exactly one operator, got {'doc_type': 'experience', 'success': False} in query.
```

### Root Cause
ChromaDB's `where` clause requires exactly one top-level operator. When `filter_dict` contains more than one key (e.g. `{"doc_type": "experience", "success": False}`), ChromaDB rejects the query.

The `VectorStore.query()` method passes `filter_dict` directly to `collection.query(where=filter_dict)` without normalizing multi-key dictionaries into ChromaDB's `$and` format.

### Affected Callers
| File | Line | Filter |
|------|------|--------|
| `skyn3t/agents/code_agent.py` | 961 | `{"doc_type": "experience", "success": False}` |
| `skyn3t/web/app.py` | 1554 | `{"doc_type": "experience"}` (single key — safe) |

### Recommended Fix
Add a `_normalize_where()` helper in `skyn3t/rag/vector_store.py` that auto-wraps multi-key dicts:

```python
def _normalize_where(filter_dict):
    if filter_dict is None or len(filter_dict) <= 1:
        return filter_dict
    return {"$and": [{k: v} for k, v in filter_dict.items()]}
```

Then apply it before passing to `collection.query(..., where=_normalize_where(filter_dict))`.

## 2. LLM Backend Timeout Failures

**Severity:** High — agents cannot complete tasks  
**Backends affected:** `kimi_cli`, `copilot_cli`, `claude_cli`

### Symptom
Repeated log entries:
```
llm complete failed; caller=code_agent backend=kimi_cli error=kimi cli failed
llm complete failed; caller=code_agent backend=copilot_cli error=idle timeout (180s no output)
llm complete failed; caller=code_critique_fix backend=claude_cli error=idle timeout (180s no output)
```

### Observations
- Timeouts are not isolated to one backend; all three CLI adapters hit idle timeouts.
- Some Kimi failures include session resume UUIDs, suggesting the CLI process starts but hangs.
- This may be caused by:
  1. CLI processes waiting on user input or 2FA
  2. Network / API rate limiting
  3. The `180s` idle timeout being too aggressive for long codegen tasks
  4. Backend credential or configuration issues

### Recommended Next Steps
- Check `.env` for `KIMI_API_KEY`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY` validity
- Verify CLI binaries (`kimi`, `copilot`, `claude`) are in `$PATH` and authenticated
- Consider increasing `idle_timeout` in adapter configs for heavy codegen tasks
- Add pre-flight health checks to CLI adapters before task assignment
