# SkyN3t Engineering Plan — Pre-Code Decision Document

## Executive Summary

SkyN3t is an autonomous multi-agent system that builds real, runnable apps end-to-end: it plans, generates code, verifies (build/boot/integration), packages, and ships scaffolds — and it does this unattended on a schedule, with a self-improvement "cortex" that learns from each build. The audit confirms the skeleton is genuinely alive, not cosmetic: the autonomous coordinator really runs ~21 builds/day gated on reviewer score, the build-pattern flywheel (record → bias → skill → inject) demonstrably fires (`data/skills/node-winning-shape.md` has success_count=123), and a 165MB experience RAG feeds real failure-recall into generation. The orchestrator core, event bus, web API surface, and dashboard SPA are well-engineered and most prior-reported bugs are genuinely fixed. **But real health is "impressive engine, several severed driveshafts."** Two classes of defect dominate: (1) silent correctness failures that make work look done while it is hollow — the worktree path delivers an empty scaffold on *every* autonomous build, post-code "retry" bails dead-end while claiming to retry, and verifiers pass apps that cannot actually function; and (2) self-learning loops captured-but-never-consumed — lesson attribution, skill grading, and stack-bucketing are wired to dead ends, so a large fraction of build signal is thrown away. The four biggest levers, in order: **(A) fix the silent build-correctness bugs (worktree merge-back, retry dead-end, verifier false-passes) so output is real; (B) close the learning feedback edges so each build measurably improves the next; (C) make generated apps complete and functional (no stubs, real persistence, functional smoke tests, visual verification); (D) fix the default-on RCE / sandbox isolation before any wider exposure.**

## Roadmap

### Theme 1 — Autonomy & Self-Learning: close the broken feedback loops

**Reality check.** This is the most pleasant surprise: SkyN3t genuinely runs unattended and one compounding loop (build-pattern → bias → skill → inject) works end-to-end with live evidence (`data/build_patterns.json` 1.9MB/73 shapes; `node-winning-shape.md` 100% over 123 samples). But the highest-value feedback edges are captured-then-discarded: `data/lesson_scores.json` does not even exist after days of running, skills are never graded, and 192 failures are stranded under `stack="unknown"` where `best_shape`/`worst_shape` can never read them. **The system practices constantly but throws away most of what it learns.**

## Recommended Execution Order

### Phase 0 — Critical bug fixes + safety (must land before anything else)
**Scope.** The two Criticals (InlineBackend RCE default + worktree merge-back) and the safety/correctness Highs that make autonomous output hollow or unrecoverable: studio retry dead-end, non-atomic manifest, the two consistency silent-aborts, contract_engine wrong-package fix, the never-stop watchdog recovery, the unparseable scout schedule, and the seatbelt `(deny default)` flip + Docker timeout-kill. Also the `request_size` cap and Discord auth exemption.
**Done looks like.** `"auto"` no longer runs untrusted code in-process (opt-in only); autonomous builds deliver a populated, verified `scaffold/`; a mid-write crash never corrupts a manifest; the consistency engine never silently no-ops; the watchdog provably restarts an exception-killed loop; the scout schedule fires; the seatbelt denies by default and Docker timeouts kill the container. New regression tests for each.

### Phase 1 — Quick high-impact wins
**Scope.** Low/medium-effort Highs with outsized payoff: the `xiaomi/` failover strip, `stdbuf` `shutil.which` guard, LLMClient `aclose` leak, backfill sanitization, the no-semicolon import regex, the TracesPage contract fix, the `/reject` parser, the editor-off-loop fix, and the Python chunker. Add the `tsc --noEmit` CI gate.
**Done looks like.** Transient OpenRouter errors no longer collapse the most visual files to stubs; CLI completions work on minimal hosts; no FD leak; backfilled files are valid; consistency catches no-semicolon scaffolds; Traces page works; REPL approval/reject are correct; `.py` ingest works. Each shipped with a focused test.

### Phase 2 — Autonomy + self-learning depth
**Scope.** Theme 1: unify stack detection (+ backfill), revive lesson attribution and route lessons into prompts, make skills self-grading + accumulate, RAG recall observability, GitHub doc ingestion, planner consults scoreboard, and the "each build improves the next" metric + integration test. Pattern-graduation is the stretch goal.
**Done looks like.** `lesson_scores.json` populates and influences injection; skill scores move with real outcomes and `curate()` can demote harmful skills; failures and successes share stack buckets; a measured rolling first-attempt success rate (per stack) trends upward and is asserted by an integration test, not just instantiation.

### Phase 3 — App quality + world-class UI
**Scope.** Themes 2 & 3 together (they share `runner.py`/`code_agent.py`/`build_verifier.py`): stub hard-gate, real persistence backend template, functional CRUD smoke test, entrypoint-imports-planned-components, code-extraction hardening, generated tests + test gate, then the token-contract enforcement, CSS-coverage gate, primitive library, frontend-design methodology, and the screenshot-based visual gate. Packaging (PWA/desktop/mobile) and the auto-* clarification fix ride here.
**Done looks like.** No project ships a stub entrypoint; data-owning apps actually persist; verification proves a CRUD round-trip and a styled, non-overflowing UI (visual score gated); generated apps consume `--brand-*` tokens and ship tests; the per-run completeness scoreboard trends to zero stub/orphan/dead-route counts.

### Phase 4 — Dashboard
**Scope.** Theme 4: shared `SwarmProvider` first (foundation), then live Studio streaming + the Cortex brain visualization + memory-layers view, the Settings/Install wizard, charts/sparklines, the living-pulse Overview, and the design-system reconciliation.
**Done looks like.** One app-wide live socket; operators can watch a build stream stage-by-stage with agent thoughts; the Cortex page shows a live agent graph + accumulating memory; CLIs/keys/integrations are configurable from the UI; trends render as charts. The dashboard feels like a living brain, not a polling panel.

## Risks & Open Questions (owner must decide)

1. **Sandbox default policy.** Phase 0 makes `"auto"` refuse in-process exec for untrusted code. **Decide:** is Docker a hard requirement on every host that runs CodeAgent/`/api/exec` (cleanest), or do we ship a hardened-subprocess fallback (rlimits, dropped privs) for Docker-less machines? This gates how aggressive the refusal can be. Also decide whether `/api/exec` should exist at all in production.
2. **Web auth + exposure model.** `web_token` defaults to None (loopback-only). **Decide:** is the dashboard ever exposed beyond loopback? If yes, the constant-time token compare, the request-size cap, and the Discord exemption all become must-fix-before-exposure (and a token holder currently gets host RCE via the exec paths).
3. **iOS / mobile / desktop scope.** Zero native output exists today. **Decide:** is mobile/iOS in scope now (drives the XL packaging effort — Capacitor for lowest friction, or true Swift/RN), later, or PWA-only for now? This is the single largest effort item in the plan.
4. **Which integrations first.** Backend supports skill-hub install, git-clone install, agent creation, Discord, GitHub Scout. **Decide:** ordering for the Settings/Install wizard (likely LLM keys → skills → GitHub Scout → Discord).
5. **Budget & model policy.** Live data shows `daily_spend_usd≈17` and a `daily cap reached (10)` skip. **Decide:** daily build cap + spend ceiling, the failover model ladder (and whether to keep `xiaomi/*` as the default `or_strong`/`or_ui` vendor), and whether to escalate to a stronger model on entrypoint regen retries (recommended in Theme 2). Note the daily-cap TOCTOU across fleet slots can overshoot — confirm in Phase 2.
6. **Pattern graduation governance.** Theme 1's transformational item auto-writes new templates/agents from winning patterns. **Decide:** is this always approval-gated, or allowed to auto-promote above a confidence threshold? This is where "self-improving" edges into "self-modifying."
7. **Default worktree isolation.** Once merge-back is fixed (Phase 0), **decide** whether worktree isolation should remain default-on for the fleet, given the cleanup contract (`remove_worktree` is never called → unbounded `.worktrees` growth) must also be added.
8. **Failed-project disposition.** **Decide:** auto-spawn `-retry` siblings vs. move failures to `_failed/` with a `FAILED.md` — affects what operators find in `Projects/` and how the learning loop treats abandoned work.
9. **Distributed mode.** `skyn3t/distributed` is unwired dead code carrying a double-delivery bug. **Decide:** wire it (real producer/consumer + fix the bug) or delete it to stop advertising a non-functional mode.
